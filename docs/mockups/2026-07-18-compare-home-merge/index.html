구현 전 검토 포인트 (역제안)

설계 조사 중 발견한, 확정 3건만으로는 안 풀리는 논점입니다. 구현 계획에 반영 여부를 정해야 합니다.

1

홈 성능(CWV) 회귀 위험

CompareBoard는 클라이언트 컴포넌트("use client")로 무료 폰트 3종 웹폰트를 실시간 렌더합니다. #25에서 홈 LCP 2.1s로 통과했는데, 이 무게가 홈에 얹히면 회귀할 수 있습니다. 권장: 뷰포트 진입 시 지연 렌더(lazy) 또는 dynamic import로 초기 로드에서 분리.

2

비교 섹션 위치 · 홈 길이

Hero+주간랭킹+비교+광고로 홈 스크롤이 길어집니다. 비교 섹션을 주간랭킹 아래·광고 위 full-width로 두는 안이 기본이나, 접기/축약 형태도 선택지입니다. 권장: 지금 배치(콘텐츠 우선, 광고 최하단) 유지.

3

모바일 탭바 "홈 vs 비교" 중복

모바일 탭바의 "홈"과 "비교"가 둘 다 홈으로 갑니다(usePathname은 해시를 무시 → 비교 탭 활성표시 불가 + 목적지 중복). 권장: 데스크톱 헤더는 /#compare 앵커 유지, 모바일 탭바에서는 "비교" 탭 제거.

4

앵커 스크롤 가림 + 크로스 페이지 이동

sticky 헤더가 #compare 섹션 상단을 가리지 않도록 scroll-margin-top이 필요합니다. 또 /fonts 등 타 페이지에서 /#compare 클릭 시 "홈 로드 후 앵커 스크롤"이 되는지 이 Next.js(정적 export) 문서로 확인해야 합니다. 권장: scroll-margin 적용 + 구현 시 문서 확인.

5

404 안내 페이지 활용

not-found.tsx가 이미 존재합니다. /compare 제거 후 옛 링크로 들어온 방문자에게 "비교는 홈으로 이동했습니다 → 홈 바로가기"를 안내하면 맨 404보다 이탈이 줍니다. 권장: not-found에 홈/비교 안내 링크 추가(소폭).
