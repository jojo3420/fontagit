### 종합 평가
- 전체 점수: 7/10
- 한줄 요약: 방향은 맞지만, “빌드타임 materialize + SSG 재빌드 + 눈누 재크롤링” 운영 흐름이 아직 너무 추상적이라 구현 계획으로 바로 넘기기엔 위험합니다.

### 항목별 리뷰
| 관점 | 평가 | 상세 |
|------|------|------|
| 완성도 | 주의 | 문제, 목표, 데이터 근거, 단계 전략은 좋습니다. 다만 재빌드 트리거, 실패 복구, 중복/정렬 규칙, prod 검증 계획이 부족합니다. |
| 누락 항목 | 주의 | 자동 컬렉션 삭제/갱신 정책, collection_items 동기화 방식, 빈 컬렉션 처리, 태그 품질 등급, 크롤링 실패 시 대안이 빠져 있습니다. |
| 일관성 | 양호 | 0단계 → B단계 → A단계 흐름은 일관됩니다. 다만 “B단계 즉시”와 “0단계 정비 후” 표현이 약간 충돌합니다. |
| 실현 가능성 | 주의 | 기술적으로 가능하지만 SSG 재빌드, DB materialize, 눈누 재크롤링은 운영 리스크가 큽니다. 설계만으로는 실패 경로가 덜 보입니다. |

### 구체적 피드백
1. [Blocker] SSG 재빌드 트리거가 너무 막연합니다.  
   “파이프라인 끝단에서 재빌드 트리거”라고 되어 있지만, 누가 어떤 조건에서 어떤 환경을 재빌드하는지 없습니다. `develop`, `main`, production 배포까지 구분해야 합니다.

2. [Blocker] auto 컬렉션 materialize 방식의 실패 처리가 없습니다.  
   rule 평가 중 실패하면 기존 `collection_items`를 유지할지, 비울지, 롤백할지 정해야 합니다. 잘못하면 컬렉션 페이지가 빈 페이지로 정적 생성됩니다.

3. [Blocker] 눈누 재크롤링은 핵심 의존성인데 대안이 약합니다.  
   robots.txt, rate limit, HTML 구조 변경을 리스크로 적었지만 “못 가져오면 어떻게 할지”가 없습니다. 최소한 기존 데이터 유지, 수동 CSV 보강, 부분 enrich 허용 같은 fallback이 필요합니다.

4. `rule jsonb` 구조가 너무 느슨합니다.  
   예시는 있지만 허용 가능한 `axis`, `op`, `value` 목록이 없습니다. 이대로면 잘못된 rule이 DB에 들어가도 빌드 때 깨집니다.

5. 자동 컬렉션의 정렬 기준이 없습니다.  
   컬렉션 안 폰트를 어떤 순서로 보여줄지 정해야 합니다. 예: Tier A 우선, 최신순, 이름순, 인기도순, 대표 굵기 보유 우선.

6. 컬렉션 중복 정책이 부족합니다.  
   auto+editorial 중복은 허용한다고 했지만, 같은 자동 컬렉션끼리 거의 같은 결과가 나올 때의 처리 기준이 없습니다.

7. `noindex=true`와 “노출 제한”이 섞여 있습니다.  
   검색엔진 색인 제외와 사이트 내부 노출 제외는 다릅니다. N 미만이면 내부 목록에서도 숨길지, 상세 접근은 허용할지 분리해야 합니다.

8. prod=dev 볼륨 가정이 미해결이면 출시 판단이 불안합니다.  
   dev 기준 1,240종 실측은 좋지만, 실제 production 데이터와 다르면 컬렉션 수, 임계값 N, SEO 전략이 흔들립니다.

9. B단계 컬렉션 후보가 아직 구체적이지 않습니다.  
   “OFL/Tier A/명조/장식/굵기”라고만 되어 있어 구현자가 바로 rule을 만들기 어렵습니다. 첫 10개 목록과 rule이 필요합니다.

10. `/fonts` 섹션 복구와 `/collections` 확장이 강하게 묶여 있습니다.  
   이 연결은 타당하지만, 범위가 커집니다. 0단계를 먼저 끝내지 못하면 B단계도 밀립니다.

11. 태그 품질 관리가 부족합니다.  
   10% 감사는 좋지만, 태그별 신뢰도나 출처 구분이 없습니다. “눈누 원본 태그”, “자동 추론 태그”, “사람 검수 태그”를 나누는 편이 안전합니다.

12. SEO description 자동 생성은 좋지만 얇은 문구 양산 위험이 있습니다.  
   같은 템플릿이 반복되면 검색 품질이 낮아 보일 수 있습니다. 컬렉션별 고유 설명 규칙이 필요합니다.

### 개선 제안
1. 우선 `0단계`와 `B단계`를 분리하세요.  
   먼저 굵기/태그 보강 파이프라인을 끝내고, 그 결과로 만들 수 있는 컬렉션만 B단계에 넣는 게 안전합니다.

2. 자동 컬렉션 rule 스키마를 명확히 정의하세요.  
   예: `axis`, `op`, `value`, `minItems`, `sort`, `limit`, `noindexBelow`를 고정 필드로 두세요.

3. materialize는 “임시 테이블 생성 → 검증 → 교체” 방식이 좋습니다.  
   실패 시 기존 `collection_items`를 유지해야 정적 페이지가 망가지지 않습니다.

4. 첫 B단계 컬렉션 8~12개를 문서에 박아두세요.  
   예: `OFL`, `Tier A`, `명조`, `장식`, `손글씨`, `다양한 굵기`, `가변 폰트`, `영문 지원`. 각 rule과 예상 개수를 같이 적어야 합니다.

5. prod 데이터 검증을 writing-plans의 첫 작업으로 넣으세요.  
   dev와 prod의 `fonts`, `collections`, `collection_items`, `weights`, `tags` 차이를 먼저 확인해야 합니다.

6. 눈누 재크롤링 fallback을 추가하세요.  
   크롤링 실패 시 기존 데이터 유지, 실패 URL 기록, 재시도 제한, 수동 보강 파일 허용 정도는 필요합니다.

7. `/collections`와 `/fonts`의 역할을 UI 문구 수준까지 정하세요.  
   지금은 전략은 좋지만 사용자가 실제로 어떻게 구분해서 쓰는지 약합니다. 컬렉션은 “테마별 묶음”, 폰트는 “전체 탐색/필터”로 고정하는 식이 좋습니다.
