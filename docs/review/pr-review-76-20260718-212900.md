# PR #76 Claude Code 리뷰 결과

## 결론

- 최초 평가: **8.5/10**, Critical 0건, High 3건
- 추가 검토: hostname 끝 점을 이용한 동일 origin 우회 High 1건 발견
- 최종 판정: High 4건 모두 수정 완료
- 검증: pipeline `171 passed`, Ruff 통과, scoped mypy 통과, 로컬 PostgreSQL SQL `ALL PASS`
- 원격 dev/prod migration과 실제 데이터 쓰기는 실행하지 않음

## 수정한 High 이슈

1. **PostgREST 역할 판정**
   - 최신 `request.jwt.claims` JSON에서 `service_role`을 확인한다.
   - claims가 없을 때만 구형 `request.jwt.claim.role`을 사용한다.
   - anon, 잘못된 JSON, 다른 역할은 모두 차단한다.

2. **자체 호스팅 Supabase 지원**
   - `.supabase.co` 주소만 허용하던 제한을 제거했다.
   - dev/prod의 HTTPS origin(주소와 포트)을 비교해 같은 서버면 차단한다.
   - 자체 호스팅 dev는 `SUPABASE_ALLOWED_DEV_ORIGINS`에 명시된 주소만 허용한다.

3. **prod 비밀키 노출 축소**
   - dev 감사 실행에 prod service key가 필요하던 조건을 제거했다.
   - 비밀이 아닌 prod URL만 사용해 dev/prod 오접속을 막는다.

4. **동일 호스트 점 표기 우회 차단**
   - `example.com.`과 `example.com`을 같은 hostname으로 정규화한다.
   - 같은 dev/prod 서버를 다른 주소처럼 적어 분리 검사를 우회할 수 없게 했다.

추가로 bootstrap RPC의 무의미한 row count 코드를 실제 `1건 삽입` 검증으로 바꿨다.

## Claude와 Codex가 함께 확인한 강점

- 다운로드 관찰기는 내부 IP, DNS 변경, 위험한 redirect, 과도한 응답 크기를 차단한다.
- manifest는 승인 근거, 이전 값, SHA-256을 모두 확인한 뒤 한 트랜잭션으로 적용한다.
- Noonnu 단독 정보는 자동 확정하지 않고 사람 검수 대상으로만 저장한다.
- 예전 Noonnu import/enrich/review/publish 경로가 공개 `fonts` 값을 직접 덮어쓰지 못하도록 차단했다.
- prod 적용은 승인 ID, 승인 SHA, 확인 SHA, 기능 플래그, 대화형 확인을 모두 요구한다.

## 보류한 Medium 항목

- 같은 forward manifest 재실행은 0건 성공이 아니라 stale 값으로 안전하게 거부된다. 운영 문서 표현을 맞춰야 한다.
- 원문 보관 정책을 켤 경우 `raw_retention_allowed` 저장 위치를 먼저 설계해야 한다.
- weights 값 범위 검증은 메타데이터 보정 단계에서 추가한다.
