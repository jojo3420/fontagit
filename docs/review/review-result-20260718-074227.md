## 2. Codex 크로스 리뷰

### 종합 소견

Claude가 찾은 굵기 축 문제는 실제 CSS와 일치했고 계획에 반영했다. 반면 테스트 환경, null 입력, 테스트 추가 제안 일부는 현재 저장소의 실행 결과·DB 제약·사용자 테스트 원칙과 맞지 않아 제외했다.

### 항목별 판정

| # | Claude 피드백 | 판정 | Codex 의견 |
|---|-------------|------|------------|
| 1 | 테스트·빌드 env 기준선 | 부분 동의 | `pnpm test`는 수정 전 172개 전체 통과했다. `tsc`의 기존 오류만 기준선 대조를 유지한다. |
| 2 | 운영 CSP 확인 | 동의 | 2026-07-18 `fontagit.com` 응답에는 CSP가 없다. 향후 CSP 추가 시 필요한 Google 도메인을 Plan에 기록했다. |
| 3 | 400/700 weight 축 | 동의 | 카드 CSS는 400, 상세 견본 CSS는 700이다. `availableWeights`에 존재하는 400/700을 CSS2 URL에 포함하도록 변경했다. |
| 4 | `nameEn` null 방어 | 동의하지 않음 | DB `name_en`은 `NOT NULL`, 웹 타입도 `string`이다. Tier A는 파이프라인이 Google API `raw.family`를 그대로 저장한다. 불변식만 Plan에 명시했다. |
| 5 | SpecimenBox·Hook lint | 부분 동의 | 기존 테스트가 이미 `font`를 전달하므로 prop 제거만 맞다. 불필요한 `useMemo`는 제거해 Hook 의존성 복잡도를 낮췄다. |
| 6 | mapper strict 비교 | 동의하지 않음 | 기존 mapper 테스트에 `toStrictEqual`이 없고 개별 필드만 검사한다. |
| 7 | UI 회귀 테스트 추가 | 동의하지 않음 | 사용자 규칙상 단순 UI 래퍼 테스트를 추가하지 않는다. 핵심 정책 3건을 자동화하고 실제 FontFace·실패 폴백은 브라우저로 직접 검증한다. |

### 동의하는 핵심 피드백

- 상세 견본의 700 굵기를 실제 폰트 파일로 요청해야 한다.
- 운영 CSP와 외부 Google 도메인의 관계를 문서화해야 한다.
- 불필요한 Hook 최적화보다 단순한 순수 함수 호출이 lint와 유지보수에 안전하다.

### 동의하지 않는 피드백

- 테스트 환경 변수 문제는 현재 저장소에서 재현되지 않았다. 수정 전 전체 172개 테스트가 통과했다.
- `nameEn` null은 DB와 TypeScript 계약이 모두 차단한다.
- UI 래퍼 E2E를 영구 추가하는 것은 현재 프로젝트의 테스트 제한과 맞지 않는다.

### Claude가 놓친 추가 관점

- Google Fonts CSS2는 `URLSearchParams`가 축 문자를 percent-encoding해도 정상 200 응답을 반환하는지 직접 확인했으며 Orbitron 400·700 `@font-face`가 모두 제공됐다.
- Tier B를 Google에 보내지 않는 구분은 폰트 이름 추측이 아니라 DB `source_tier`를 사용해야 한다.
