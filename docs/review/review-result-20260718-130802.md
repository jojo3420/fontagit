## 구체적 피드백

1. **[치명·매칭] prod `font_sources` backfill의 진실 공급원이 미정의.** 5.2는 `(provider, provider_record_id)`가 "환경 안에서 unique"라 규정하지만, dev→prod 매칭이 성립하려면 **동일 `provider_record_id`가 prod에도 존재**해야 한다. 12단계 8은 "prod 안정 출처 키 backfill"을 지시할 뿐, prod 각 행에 눈누 페이지번호·구글 family key를 **무엇으로 다시 채우는지** 없다. 재스크랩 후 이름으로 매칭해 채운다면 정확히 회피하려던 이름 매칭으로 되돌아가는 순환이다. → prod 행에 provider_record_id를 유도할 수 있는 기존 컬럼(예: `official_url`/slug/시드 잔재)을 명시하고, 유도 불가 행은 자동 backfill 금지·검수로 보내는 규칙 필요.

2. **[치명·정합] prod `foundry`가 전량 공란인데 매칭 precondition에 "제작사"를 포함.** 2절은 "prod 1,240종 모두 foundry가 비어 있다", 5.2·6.3은 매칭 보조키로 "제작사"를 요구한다. backfill 이전 prod에는 제작사가 없어 이 precondition이 **전 행에서 실패**하거나 무력화된다. → 1단계에서는 제작사를 매칭키에서 제외하거나, foundry backfill을 안정키 연결보다 먼저 수행하는 순서를 명시.

3. **[치명·신뢰] 라이선스 6개 항목 추출 방식이 비어 있음.** 공개 요약은 "결정론적 템플릿, LLM 자유서술 금지"로 잘 방어했지만, 그 **입력이 되는 6개 구조화 값(`allow_*`, `attribution_requirement`) 자체를 무엇이 추출하는지** 규정이 없다. LLM이 원문에서 이 값을 뽑는다면 hallucination이 결정론 템플릿의 상류로 그대로 유입된다. → 6항목은 (a) 승인된 라이선스 정형 매핑(예: OFL/공유마당 유형표) 또는 (b) 사람 검수 확정만 verified로 인정하고, LLM 추출은 무조건 needs_review로 강등한다고 명시.

4. **[보안·SSRF] DNS 재검사는 있으나 IP 고정(pinning)이 빠져 rebinding TOCTOU 잔존.** 7절은 "모든 리다이렉트 단계에서 DNS 재검사"까지 했으나, 검사 시점과 실제 연결 시점 사이 재resolve로 사설 IP가 반환되는 경합을 막으려면 **검증한 IP로 직접 연결(연결 IP 고정)**이 필요. → 검증 통과한 IP에 핀 고정 후 Host 헤더로 요청하는 방식 추가.

5. **[정합] weekly 무상태 검사와 "24시간·2회 404 → broken" 규칙의 연결 미비.** 10절 예약 작업은 DB 쓰기 키가 없어 상태를 보존하지 못한다. 그런데 7절 broken 판정은 "최초 확인일로부터 24시간 지난 독립 실행에서도 404"라는 **직전 상태 참조**를 전제한다. 첫 404 시각을 어디에 저장하고 다음 실행이 어떻게 읽는지(사람 import 단계가 `download_status=needs_review`+`download_checked_at`을 이어받아 2회차를 판정) 경로를 명시해야 규칙이 실제로 작동한다.

6. **[누락·되돌리기] 역방향 manifest의 복원 정확성 테스트가 없다.** 12절은 "역방향 manifest도 부분 복원 불가"라 못박았지만, 11절 테스트에 **되돌리기가 실제로 이전 값을 재현하는지** 검증이 빠졌다. all-or-0 적용만큼 롤백 정확성도 데이터 손실과 직결되므로 픽스처 기반 왕복(적용→역적용→원상 복구) 테스트 1개 필요.

7. **[누락·무결성] `download_evidence_id`/`license_evidence_id`와 snapshot 간 FK·정합 규칙 부재.** 검증값이 참조하는 근거 스냅샷이 실제 그 필드의 출처와 일치하는지 보장이 없다. apply RPC 검사 항목에 "evidence_id가 존재하고, 그 스냅샷의 provider·자료 종류가 확정 필드와 일치"를 추가하고 DB FK로 강제 권장.

8. **[실현성·병목] 검수 큐 물량과 해소율(yield) 목표가 없다.** Tier B 소형 폰트 다수는 공식 제작사 페이지가 없어 상당수가 needs_review로 귀결될 가능성이 크다. 14절 완료 기준은 "모든 폰트가 상태를 가진다"라 **미확인 100%여도 형식상 완료**된다. 프로세스 완결성과 데이터 품질 산출이 분리돼 있음. → "verified 최소 비율" 또는 "미확인 허용 상한"을 명시하고, 단독 운영자 기준 예상 검수 건수·소요를 리스크로 기재.

9. **[보안·오염] 도메인 레지스트리의 초기 부트스트랩 경로가 무방비.** 6.3은 "사람이 승인한 레지스트리"만 자동 근거로 인정해 잘 막았지만, **그 레지스트리 후보를 사람이 어떻게 발굴하는지**(임의 웹검색 결과가 승인 직전 단계로 유입될 수 있음)가 없다. → 후보 발굴 시에도 출처(제작사 공식 표기·공공기관 공식 도메인 목록)를 근거로 남기고, 웹검색 단독 발굴은 별도 라벨로 검수한다고 명시.

10. **[정합·의미손실] `conditional`의 boolean dual-write.** `allow_commercial='conditional'` → `is_commercial_free=false` 매핑은 보수적이라 안전하지만, legacy 소비자에게 "상업 사용 불가"로 오독될 수 있다(실제론 조건부 허용). 전환 기간 legacy 필터가 이 폰트를 부당하게 걸러낼 수 있음. → legacy 소비자 전수 확인(5.1에 약속됨) 시 conditional 처리 영향까지 점검 대상에 포함하고, 가능하면 조기에 신규 필드로 이관.

11. **[미증명] 호출처 전수 확인이 "약속"에 머무름.** 5.1은 mapper·RPC·컬렉션·uploader·제출/검수 경로 전수 확인을 지시하나 **실제 grep 결과 목록이 문서에 없다.** 설계 승인 단계에선 허용되지만, 착수 전 legacy 3필드(`official_url`, `is_commercial_free`, `license_verified`) 호출처 실측 목록을 부록으로 첨부해야 dual-read/write 범위가 확정된다.

12. **[누락·법적] 눈누 원문 대량 내부 저장의 ToS/저작권 검토 부재.** robots.txt·UA·간격은 지켰으나, 폰트 관련 HTML 전문 저장은 스크래핑 ToS 측면 리스크가 남는다. 내부 증거 목적·비공개·비렌더링으로 완화되나 한 줄이라도 근거를 남기는 편이 안전.

## 개선 제안

**우선순위 1 (착수 전 필수 해소)**
- 피드백 1·2: prod 안정키 backfill의 진실 공급원과 순서를 확정. 유도 불가 행은 자동 backfill 금지·검수. 이게 없으면 매칭 정확도 자체가 보장되지 않는다.
- 피드백 3: 라이선스 6항목의 verified 인정 조건을 "정형 매핑 또는 사람 확정"으로 못박고 LLM 추출은 needs_review 강등.

**우선순위 2 (구현 전 반영 권장)**
- 피드백 4(IP 고정), 5(weekly↔broken 상태 인계), 7(evidence FK)로 보안·판정·무결성 규칙의 실행 경로를 닫는다.
- 피드백 6: 되돌리기 왕복 테스트를 11절 테스트 목록에 추가(데이터 손실 커버리지는 생략 불가 영역).

**우선순위 3 (운영 안정화)**
- 피드백 8: 미확인 허용 상한 또는 최소 verified 비율을 완료 기준에 추가하고, 예상 검수 물량을 리스크로 기재. 병렬 검수 도구(일괄 승인 UI)도 고려.
- 피드백 9·10·11·12: 레지스트리 부트스트랩 근거화, conditional 영향 점검, 호출처 grep 부록, 스크래핑 ToS 한 줄 근거.

**총평:** 적대적으로 가장 위험한 "출처 오염·SSRF·부분 적용·권한 고정"은 이미 잘 막혔다. 남은 리스크는 대부분 **"매칭·추출의 상류가 미정의"**라는 한 축에 몰려 있으니, 우선순위 1의 두 항목만 확정하면 실행 가능성이 크게 올라간다.

## 2. Codex 크로스 리뷰

### 종합 소견

Claude는 자체 적대적 리뷰에서 이미 보강한 출처 경계·원자적 적용·SSRF 방어를 정확히 확인했고, 그 상류인 안정키 bootstrap과 라이선스 구조화 값의 신뢰 모델이 비어 있다는 핵심 구멍을 추가로 찾았다. 12건 중 10건은 그대로 반영하고, 2건은 이 프로젝트의 실제 실행 방식에 맞게 조정한다.

### 동의하는 핵심 피드백

1. 안정키 bootstrap을 별도 단계로 분리하지 않으면 slug 단독 매칭으로 되돌아가므로 구현 시작을 막아야 한다.
2. 라이선스 구조화 값은 정형 매핑·사람 승인 없이는 verified가 될 수 없도록 해야 한다.
3. prod 적용보다 먼저 dual-read consumer를 배포하지 않으면 conditional 값이 유료로 오표시될 수 있다.

### 동의하지 않는 피드백

완전히 기각한 항목은 없다. 다만 직접 IP 연결을 유일한 SSRF 해법으로 고정하는 것과 근거 없는 verified 최소 비율 설정은 과잉이다. 각각 TLS를 보존하는 pinned transport 또는 네트워크 격리, 100% 분류와 명시적 미확인 승인으로 조정한다.

### Claude가 놓친 추가 관점

- 자체 리뷰에서 기존 import·enrich·uploader가 검증 필드를 다시 덮지 못하도록 audit apply RPC를 단일 writer로 제한했다. 구현 계획에서도 이 경계를 별도 작업으로 유지해야 한다.
- 호환 배포는 단순히 호출처를 고치는 것뿐 아니라 additive schema → dual-read consumer → manifest → CHECK 강화 순서여야 한다.
- 예약 작업 실패나 새 오류 발견이 조용히 묻히지 않도록 실패 알림과 결과 artifact 존재 검사를 운영 게이트로 둬야 한다.

## 3. 최종 권고사항

### 즉시 반영 (Must)

1. Tier A upstream snapshot과 Tier B 원본 seed/proposal을 이용한 1회성 안정키 bootstrap manifest 및 불일치 검수 규칙
2. 정형 라이선스·승인 템플릿·사람 검수만 verified로 인정하는 6개 항목 신뢰 모델
3. evidence FK와 provider·문서 종류 일치 검증
4. 예약 observation의 dev 누적 판정과 24시간 2회 실패 규칙
5. manifest 적용→역적용→원상복구 왕복 테스트
6. additive schema → dual-read consumer → manifest → CHECK 강화 배포 순서
7. legacy 소비자 실측 부록과 단일 writer 경계

### 검토 후 반영 (Should)

1. pinned resolver/transport 또는 네트워크 격리 실행으로 DNS rebinding 방어
2. 50건 pilot 후 needs_review가 10%를 넘으면 상위 제작사 템플릿부터 규칙을 보강
3. 도메인 레지스트리 후보별 승인 근거 snapshot
4. 전체 수집 전 robots·이용약관 저장 범위 확인

### 참고 (Nice to have)

1. 예약 작업 실패·빈 artifact 알림
2. 동일 본문 조건부 요청(ETag·Last-Modified)으로 외부 트래픽 절감
