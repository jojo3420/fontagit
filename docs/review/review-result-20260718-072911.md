### 구체적 피드백

1. 설계 완료 조건 중 sitemap URL 집합과 indexable canonical 집합 일치는 Task 4A에만 있고 아직 구현되지 않았다.
2. Task 4의 전체 검증 완료 표시가 뒤늦게 발견된 Task 4A 결함과 충돌한다. Task 4를 부분 검증으로 낮춰 표현해야 한다.
3. 현재 배포 스크립트에는 branch·HEAD·clean 검사 없이 `--commit-dirty=true`가 남아 있어 잘못된 산출물이 운영에 갈 수 있다.
4. Supabase/PostgREST 행 수 상한을 넘으면 slug 조회가 조용히 잘릴 수 있다. 명시적 페이지네이션과 exact count 대조가 필요하다.
5. `robots.ts`가 계획의 수정·검증 파일에 없어 BASE_URL 변경의 추적성이 약하다.
6. 사고 원인이었던 `NEXT_PUBLIC_BASE_URL` 읽기 자체를 제거한다는 내용이 계획에 명확히 적혀야 한다.
7. 운영 빌드가 어떤 Supabase 설정을 사용하는지와 개발 DB 혼입 방어가 문서에 필요하다.
8. 과거 Cloudflare 배포로의 롤백 방지는 기술적 강제가 아니라 운영 절차임을 잔존 리스크로 인정해야 한다.
9. Search Console 제출과 실제 색인, TBT와 INP를 분리한 점은 강점이다. 다만 GA4 외부 수행은 운영 검증 전 상태임을 표시해야 한다.
10. 보호 파일 목록이 현재 워킹트리와 달라질 수 있고, develop→main 승격 흐름과 www 리다이렉트 유지 검증이 빠져 있다.

Claude의 핵심 지적은 모두 실제 코드와 문서에서 확인됐다. 특히 1,000행 상한과 Task 4 완료 과표시는 자체 리뷰가 놓친 중요한 결함이었다. 다만 운영 DB가 특정 연결 방식이라는 추정은 문서 증거가 부족해 일반적인 `.env.production` 출처 검증으로 좁혀 반영했다.

### 동의하는 핵심 피드백

1. Task 4A는 선택 보강이 아니라 운영 배포의 하드 선행조건이다.
2. sitemap 집합 대조만으로는 같은 방식으로 잘린 데이터 집합을 놓칠 수 있어 exact count·페이지네이션이 필요하다.
3. 배포 스크립트가 배포 출처를 자동으로 검증하지 않으면 문서 승인 게이트만으로는 부족하다.

### 동의하지 않는 피드백

- Claude가 언급한 특정 prod DB 연결 방식은 이번 두 문서와 현재 출력만으로 확정할 수 없다. 따라서 특정 호스트·터널을 설계에 박지 않고, 실제 `.env.production`이 존재하고 개발 URL과 다르며 비밀값을 출력하지 않는 조건만 반영했다.

### Claude가 놓친 추가 관점

- 운영 폰트 메타 설명에 제작사 누락으로 `" 제작 서체"`가 실제 노출됐다. Task 4A에서 확인 가능한 값만 조합하도록 별도 수정해야 한다.
- 운영 전수 검증은 sitemap URL의 200뿐 아니라 자기참조 canonical까지 확인해야 한다.
