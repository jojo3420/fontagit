## 유지한 지적

### 🔴 배포 출처가 고정되지 않음

- 증거: `scripts/deploy.sh:37-43`은 현재 브랜치·원격 SHA·dirty 상태를 확인하지 않고 `--commit-dirty=true`로 운영 배포한다.
- 위험: SEO 수정 외의 사용자 변경이나 다른 기능 브랜치가 같은 `out/`에 섞여 배포될 수 있다.
- 반영: 설계에 clean `origin/main` 전용 worktree와 사용자 승인 게이트를 추가하고, 계획 Task 4A에 스크립트 수정 작업을 넣었다.

### 🔴 설계의 전수 검증 약속과 실제 검증기가 다름

- 증거: `apps/web/scripts/verify-seo-output.mjs:62-71`은 폰트·컬렉션이 각각 1개 이상인지만 확인한다. 부분 누락·HTML canonical 불일치는 검출하지 못한다.
- 위험: 130개 중 일부만 sitemap에 있어도 배포 검증을 통과할 수 있다.
- 반영: sitemap 집합과 빌드 HTML의 indexable 자기참조 canonical 집합을 정확히 비교하도록 설계·Task 4A를 보강했다.

### 🔴 배포 성공과 운영 검증 성공이 분리되지 않음

- 증거: `scripts/deploy.sh:43-51`은 스모크 체크 실패를 허용하고 마지막에 완료를 출력한다.
- 위험: Cloudflare 업로드만 성공한 상태를 색인 복구 완료로 오판할 수 있다.
- 반영: 배포·운영 전수 검증·sitemap 제출·실제 색인을 서로 다른 상태로 문서화했다.

### 🟡 빈 제작사로 메타 설명이 깨짐

- 증거: `apps/web/app/fonts/[slug]/page.tsx:40-43`은 제작사 값을 그대로 문장 앞에 붙인다. 운영 본고딕에서 `" 제작 서체"`가 확인됐다.
- 반영: 확인 가능한 조각만 조합하도록 설계와 Task 4A에 추가했다.

### 🟡 Search Console 제출을 실제 색인과 혼동할 수 있음

- 증거: 기존 완료 조건은 제출 성공 또는 처리 대기만 기록해도 전체 작업 완료처럼 읽혔다.
- 반영: 제출·URL 실시간 검사·실제 색인 수를 단계별로 구분했다.

### 🟡 검색 순위 작업과 기술 색인 복구의 경계가 없음

- 위험: sitemap 복구만으로 검색 유입 최적화까지 끝났다고 오판하거나, 반대로 같은 작업에 키워드·콘텐츠 확장을 과도하게 섞을 수 있다.
- 반영: P0 기술 복구와 후속 검색 노출 작업을 분리하고, 대량 유사 페이지 금지와 INP 현장 측정 조건을 명시했다.

### 🔴 1,000행 이후 slug 조회가 조용히 잘릴 수 있음

- 증거: `getAllSlugs()`, `getPublishedSlugs()`, `getAllCollectionSlugs()`는 범위 지정·exact count 없이 단일 Supabase 조회를 사용한다.
- 위험: 카탈로그가 서버 행 수 상한을 넘으면 sitemap과 정적 페이지가 일부만 생성될 수 있다.
- 반영: 명시적 페이지네이션, exact count, 수집 개수 일치 검증을 Task 4A의 하드 게이트로 추가했다.
