## 구체적 피드백

1. (재배포 경로 — 가장 중요) 5장·7장의 "배포 스크립트가 빌드 결과의 sitemap.xml, robots.txt를 검사한다"는 가드가 **어디서 실행되는지**가 없다. Cloudflare Pages가 git push로 자동 빌드하면 로컬/CI 스크립트는 우회된다. 가드가 Pages의 빌드 커맨드 안에 들어가지 않으면, 정상 배포 경로로 잘못된 sitemap이 그대로 다시 나갈 수 있다. 이 설계의 핵심 목표(재발 방지)가 무력화되는 지점이다.

2. (롤백 위험 — 문서에 없음) 7장 "배포가 실패하면 이전 잘못된 sitemap을 성공으로 보고하지 않는다"는 보고 원칙만 있고 롤백 계획이 없다. 그런데 **직전 배포본이 곧 example.com 버전**이다. Cloudflare Pages 롤백은 이전 배포로 되돌리는 것이라, 롤백하면 버그가 그대로 복원된다. "롤백 불가, 전진 수정만 가능"임을 명시해야 한다.

3. (5장 vs 8장 불일치) 5장 검사는 "`fontagit.example.com`이 남거나 `https://fontagit.com`이 없으면 중단"이다. 이건 부분 문자열 존재 검사라 URL 하나만 맞아도 통과한다. 절반이 다른 잘못된 도메인(예: `*.pages.dev`, 오탈자 도메인)이어도 `https://fontagit.com`이 어딘가 하나 있으면 검사를 통과한다. 반면 8장 완료 조건은 "모든 `<loc>`이 `https://fontagit.com`으로 시작"이다. 가드를 8장 기준(전수 접두 검사)으로 맞춰야 한다.

4. (개수 기준값 부재) 1장은 현재 137개, 3장은 여러 페이지를 제외한다. 7장 2단계 "URL 개수를 검사"는 **무엇과 비교하는지** 기준이 없다. 제외 후 기대 개수를 명시하지 않으면 개수 검사는 사실상 무의미하다.

5. (색인 정책 — 절반 정확, 절반 공백) 4장 "robots 차단은 색인 제외 수단이 아니다"는 Google 정책상 **정확**하다. 다만 3장이 `/search/`를 sitemap에서 제외하는 의도는 "색인 원치 않음"인데, sitemap 제외만으로는 색인을 막지 못한다(내부 링크로 발견·색인됨). 그런데 4장은 noindex를 "필요하면" 붙이는 선택 사항으로 남겼다. 의도(색인 배제)와 수단(선택적 noindex)이 어긋난다. `/search/`를 정말 색인에서 빼려면 noindex는 선택이 아니라 필수다.

6. (GA4 빌드타임 env) 6장 "`NEXT_PUBLIC_GA_ID`를 운영 환경 파일에 저장, 저장소엔 커밋 안 함"은 정적 export 모델과 충돌한다. `NEXT_PUBLIC_*`는 빌드 시점에 코드에 박히므로 런타임 "환경 파일"이 아니라 **Cloudflare Pages 빌드 환경(대시보드)**에 있어야 한다. 커밋도 안 하고 빌드 환경에도 없으면 스크립트가 아예 안 실려 8장 "GA4 스크립트 로드" 조건을 만족 못 한다.

7. (GA4 중복/누락 수집 미검토) 6장은 컴포넌트 재사용과 조건부 로드까지만 다룬다. 기존에 다른 분석 도구(GTM, Cloudflare Web Analytics, 하드코딩 gtag)가 있으면 이중 집계 위험이 있고, SPA 라우트 전환 시 page_view 중복/누락(enhanced measurement와 수동 `send_page_view` 충돌) 여부도 없다. 사용자가 지적한 "GA4 중복 또는 누락"이 그대로 열려 있다.

8. (Search Console 선행 조건 누락) 7장 5단계 "sitemap 제출"은 해당 속성이 **먼저 검증(DNS TXT 또는 HTML 파일)**되어 있어야 가능하다. 검증 절차와, apex 전용 URL-prefix 속성인지 도메인 속성인지가 없다. 외부 콘솔 작업이라 코드로 검증 불가능하므로, 이 선행 단계를 절차에 넣지 않으면 5단계에서 막힌다.

9. (외부 콘솔 주장 미검증) 1장 "www→apex 301", 6장 GA4 속성 생성, Search Console 상태는 모두 외부 콘솔 사실이라 코드로 확인할 수 없다. 특히 301이 실제로 설정돼 있지 않으면 apex 대표 도메인 전제가 깨진다. 이들은 "가정"으로 라벨링하고 확인 절차(스크린샷/curl 리다이렉트 확인)를 붙여야 한다.

10. (trailing slash 대표 URL 정합) 3장 URL은 모두 후행 슬래시(`/fonts/{slug}/`)다. Next.js `trailingSlash` 설정과 각 페이지 canonical 태그가 이와 일치하지 않으면(예: 슬래시 없는 URL로 301), sitemap URL이 "대표 URL"이라는 8장 조건을 위반한다. 설정 확인이 필요하다.

11. (기존 테스트 의존 주장) 5장 "도메인 기본값과 sitemap 핵심 URL만 기존 테스트에서 검증한다"는 **기존 테스트가 실제로 그걸 커버한다는 미검증 전제**다. BASE_URL 기본값 테스트가 없으면 이 문장은 희망 사항이다. 커버 여부를 확인하거나, 없으면 최소 그 한 케이스는 추가해야 한다.

12. (공개 폰트 필터 정의 부재) 3장 "공개 폰트 상세"의 "공개" 판정 기준(초안/비공개 제외 로직)이 없다. 비공개 폰트가 sitemap에 새면 색인되어선 안 될 페이지가 노출된다.

## 개선 제안

Must (구현 전 필수)
- 가드를 Cloudflare Pages **빌드 커맨드 내부**에 배치해 자동 배포 경로가 우회하지 못하게 한다(피드백 1).
- 롤백하면 example.com 버전이 되살아남을 문서에 명시하고 "전진 수정만" 원칙을 못 박는다(피드백 2).
- 5장 검사를 8장 완료 조건과 동일한 **전수 접두 검사 + 기대 URL 개수**로 통일한다(피드백 3, 4).
- 정적 export 기준으로 `NEXT_PUBLIC_GA_ID`·`NEXT_PUBLIC_BASE_URL`을 **빌드 환경(대시보드)**에 두는 것으로 문구를 바로잡는다(피드백 6).
- Search Console 속성 검증을 sitemap 제출보다 앞선 단계로 절차에 추가한다(피드백 8).

Should
- `/search/` noindex를 "필요하면"이 아니라 필수로 확정하거나, 제외 의도 자체를 재검토한다(피드백 5).
- 기존 분석 도구 유무·SPA page_view 중복/누락 점검을 6장에 추가한다(피드백 7).
- 외부 콘솔 사실(301, GA4 속성, SC 상태)을 "가정"으로 라벨링하고 확인 증거(리다이렉트 curl, 스크린샷)를 완료 조건에 넣는다(피드백 9).
- `trailingSlash`·canonical 태그가 sitemap URL과 일치하는지 확인 항목을 추가한다(피드백 10).
- "공개 폰트/컬렉션" 판정 로직을 명시한다(피드백 12).

Nice to have
- 기존 테스트의 BASE_URL 기본값 커버 여부를 확인하고, 없으면 그 한 케이스만 추가(피드백 11).
- Search Console 속성 유형(도메인 vs URL-prefix)을 명시.

⚠️ 위 리뷰는 문서 텍스트만 근거로 했고, 코드베이스·외부 콘솔은 대조하지 않았다(요청에 따라 파일 접근·명령 실행 없음). 137개·www 301·기존 테스트 커버리지 등 문서 내 주장은 미검증 전제로 다뤘다.

## 2. Codex 크로스 리뷰

### 종합 소견

재발 방지와 sitemap 대상 정의가 약하다는 지적은 타당하다. 다만 Claude는 저장소를 보지 않았기 때문에 Cloudflare가 직접 빌드한다고 가정했고, 이 프로젝트의 실제 로컬 빌드 후 업로드 흐름과 충돌하는 의견이 있다.

### Claude가 놓친 추가 관점

- sitemap에서 제외할 보류·배포종료 페이지는 URL 자체를 삭제하면 안 된다. 기존 제품 요구사항상 상태 안내 페이지를 유지하되 sitemap 제외와 `noindex,follow`를 적용해야 한다.
- 컬렉션 상세와 정적 SEO 랜딩 페이지에는 canonical이 없다. sitemap의 대표 URL 주장과 맞추려면 canonical 메타데이터도 함께 보강해야 한다.
- 배포 검증은 단순 문자열 검색이 아니라 XML 파싱 성공, `<loc>` 중복 없음, 허용 origin 일치, HTTP 200 확인까지 나눠야 한다.

## 3. 최종 권고사항

### 즉시 반영 (Must)

1. sitemap용 slug 조회를 `published` 전용으로 분리한다.
2. `/search/`, 보류·배포종료 폰트에 `noindex,follow`를 적용한다.
3. sitemap의 모든 `<loc>` origin, 중복, 계산된 개수를 업로드 직전에 검사한다.
4. pre-fix 배포로 롤백하지 않고 검증된 배포 ID만 롤백 대상으로 기록한다.
5. 실제 도메인 기본값 회귀 테스트와 sitemap 핵심 테스트를 갱신한다.

### 검토 후 반영 (Should)

1. sitemap에 넣는 정적 페이지와 컬렉션 상세에 canonical을 추가한다.
2. GA4 향상된 측정의 브라우저 기록 변경을 활성화하고 DebugView에서 초기 로드·내부 이동이 각각 1회인지 확인한다.
3. 운영 URL 표본을 직접 요청해 200과 canonical 일치를 확인한다.

### 참고 (Nice to have)

- sitemap URL 수는 고정 숫자가 아니라 `정적 7 + 공개 폰트 + 공개 컬렉션` 계산값으로 기록한다.
