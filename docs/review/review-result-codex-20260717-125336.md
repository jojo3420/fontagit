### 종합 평가
- 전체 점수: 6.5/10
- 한줄 요약: 방향은 맞지만, **동시 요청 race condition**과 **Kong 실제 IP/라우팅 검증 누락** 때문에 그대로 구현하면 방어가 뚫릴 수 있습니다.

### 항목별 리뷰
| 관점 | 평가 | 상세 |
|------|------|------|
| 완성도 | 주의 | 목적, 위협 모델, A/B 방어, 테스트 계획은 잘 잡혔습니다. 다만 동시성, 롤백, 운영 검증 기준이 부족합니다. |
| 누락 항목 | 미흡 | 병렬 봇 요청, Kong이 실제 사용자 IP를 보는지, 기존 Supabase 인증/라우트와 충돌하는지 빠져 있습니다. |
| 일관성 | 양호 | 0007 후속, 0008/0009 재배치, 정적 SSG 제약, anon RPC 구조는 문서 안에서 대체로 일관됩니다. |
| 실현 가능성 | 주의 | DB 함수 교체는 가능하지만 현재 SQL은 동시 요청에 약합니다. Kong 설정은 실물 설정 확인 전까지 개념 수준입니다. |

### 구체적 피드백
1. [Blocker] DB rate limit이 동시 요청에 취약합니다.  
   현재는 `count(*)` 후 `insert`라서 봇이 동시에 100개 요청을 보내면 여러 요청이 모두 `20건 미만`으로 보고 통과할 수 있습니다. 이 기능의 핵심 목적이 봇 방어라서 구현 전 반드시 막아야 합니다.

2. [Blocker] Kong의 `limit_by: ip`가 실제 사용자 IP를 보는지 검증 항목이 없습니다.  
   OCI 로드밸런서, 프록시, Cloudflare 같은 것이 Kong 앞에 있으면 Kong은 모든 요청을 같은 IP로 볼 수 있습니다. 그러면 정상 사용자까지 같이 429를 맞거나, 반대로 제한이 무력화됩니다.

3. [Blocker] Kong 스니펫이 실제 Supabase Kong 설정과 충돌할 수 있습니다.  
   새 `service/routes`를 추가하면 기존 `/rest/v1` 라우트의 인증, CORS, apikey 처리와 달라질 수 있습니다. “기존 route에 플러그인을 붙일지”, “별도 route가 우선 매칭되는지”를 명확히 해야 합니다.

4. `count(*)`는 단순하지만 방어 코드로는 아쉽습니다.  
   최근 10초에 수천 건이 이미 쌓인 상황에서는 매번 count가 커질 수 있습니다. `20번째 존재 여부만 확인`하거나, 별도 카운터 테이블 방식이 더 안전합니다.

5. 테스트 계획에 병렬 요청 테스트가 없습니다.  
   현재 테스트는 순차 호출 기준입니다. 실제 공격은 병렬입니다. 최소한 “동시 50회 호출 후 20건 이하 유지” 검증이 필요합니다.

6. `font_clicks` 원본 테이블 보관 정책이 없습니다.  
   rate limit은 폭증을 줄일 뿐, 장기적으로 테이블이 계속 커지는 문제는 해결하지 않습니다. `font_click_daily`가 이미 있다면 원본 클릭 보관 기간도 정해야 합니다.

7. “조용히 무시”는 UX에는 좋지만 운영 관측이 없습니다.  
   제한이 자주 걸리는지 알 수 없으면 오탐인지 공격인지 판단하기 어렵습니다. 최소한 익명 집계 수준의 확인 방법이 필요합니다.

8. A안 검증 절차가 너무 짧습니다.  
   40회 curl만으로는 부족합니다. 429 발생, 60초 후 해제, 다른 RPC 영향 없음, 실제 DB insert 수 감소까지 확인해야 합니다.

9. 롤백 절차가 없습니다.  
   오탐이 발생하면 0008 함수를 되돌리는 SQL, Kong 플러그인 비활성화 방법이 바로 있어야 합니다.

### 개선 제안
1. 최우선: DB 함수에 폰트별 advisory lock을 추가하세요.  
   가장 단순한 보완입니다. `v_font_id` 조회 후 같은 폰트 요청을 잠깐 직렬화하면 `count 후 insert` race를 막을 수 있습니다.

2. Kong 적용 전에 실제 prod `kong.yml` 기준으로 route/auth/CORS/IP 전달 구조를 먼저 확인하세요.  
   개념 스니펫은 문서에 남기되, “그대로 붙여넣기 금지”라고 명확히 쓰는 편이 안전합니다.

3. 테스트에 병렬 호출 케이스를 추가하세요.  
   이 설계의 핵심은 봇 방어이므로 순차 테스트만 있으면 핵심 검증이 빠진 상태입니다.

4. 원본 클릭 보관 기간을 정하세요.  
   예: `font_clicks`는 30일만 보관하고, 장기 통계는 `font_click_daily`에 남기는 방식이 더 현실적입니다.

5. 운영 검증 기준을 구체화하세요.  
   “429 확인”뿐 아니라 “DB insert 수가 제한되는지”, “정상 클릭 UX가 깨지지 않는지”, “다른 Supabase RPC가 영향받지 않는지”까지 포함해야 합니다.
