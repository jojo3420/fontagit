### 종합 평가

- 전체 점수: 4/10
- 한줄 요약: 핵심 위험은 잘 찾았지만, 현재 상태는 기능 계획이 아니라 아이디어 목록에 가깝습니다. 범위·운영 주체·데이터 생명주기·완료조건을 먼저 고정해야 합니다.

### 항목별 리뷰

| 관점 | 평가 | 상세 |
|------|------|------|
| 완성도 | 미흡 | 수집·검수·게시·수정·종료·삭제의 전체 운영 흐름이 없습니다. 목록+상세 요구도 완료조건에 반영되지 않았습니다. |
| 누락 항목 | 미흡 | 중복 제거, 마감 변경, 공고 취소, 시간대, 장애 처리, 관리자 운영, 검색 노출, 보안, 삭제 요청 등이 빠졌습니다. |
| 일관성 | 주의 | 4개 카테고리를 제시했지만 완료조건은 1개만 요구합니다. “별도 도메인”이라고 하면서 개인 폰트는 #64와 경계가 겹칩니다. |
| 실현 가능성 | 주의 | 정적 사이트에서도 런타임 Supabase 조회로 구현할 수 있습니다. 다만 상세 페이지 검색 노출, 마감 상태 계산, 게시물 관리 방식에 제약이 있습니다. |

### 구체적 피드백

1. **완료조건이 지나치게 약합니다.**  
   게시물 한 건만 보여도 완료로 판정됩니다. 목록·상세·분류·정렬·오류 화면·마감 상태가 모두 검증되도록 바꿔야 합니다.

2. **목록+상세 요구가 완료조건에 없습니다.**  
   특히 `output: 'export'`에서는 새 게시물마다 `/board/[slug]` 상세 페이지를 자동 생성하기 어렵습니다. 다음 중 하나를 미리 선택해야 합니다.

   - `/board?id=123` 형태로 런타임 조회: 구현은 쉽지만 검색엔진 노출이 약함
   - 게시 시 재빌드하여 상세 페이지 생성: 검색 노출은 좋지만 운영 복잡도와 지연 증가
   - 서버·Edge Function 추가: 유연하지만 현재 정적 구조에서 범위가 커짐

3. **“게시일”의 의미가 모호합니다.**  
   원문 게시일, fontagit 등록일, 마지막 수정일을 분리해야 합니다. 최소한 `source_published_at`, `created_at`, `last_verified_at`이 필요합니다.

4. **마감 상태를 저장값 하나로 관리하면 반드시 낡습니다.**  
   `진행 중`을 저장하지 말고 마감일과 현재 시간을 비교해 계산해야 합니다. 단, 조기 종료·기간 연장·취소는 별도 상태로 덮어쓸 수 있어야 합니다.

5. **시간 기준이 없습니다.**  
   마감일이 날짜만 있는지 시각까지 있는지, `7월 31일까지`를 몇 시까지로 볼지, 기본 시간대를 한국 시간으로 볼지 정해야 합니다. 사용자 기기 시간이 틀린 경우도 고려해야 합니다.

6. **마감일이 없는 사업을 처리할 수 없습니다.**  
   상시 모집, 예산 소진 시 종료, 선착순, 일정 미정이 존재합니다. 이를 일반 날짜와 같은 방식으로 정렬하면 잘못된 안내가 됩니다.

7. **공고 변경과 취소 흐름이 없습니다.**  
   원문에서 마감 연장·자격 변경·취소가 발생했을 때 기존 게시물을 수정할지, 변경 표시를 붙일지, 종료 처리할지 정해야 합니다.

8. **데이터 생명주기가 빠졌습니다.**  
   게시물이 `수집됨 → 검수 대기 → 공개 → 수정 → 마감/보관 → 삭제`되는 흐름과 각 단계의 담당자가 필요합니다.

9. **관리자 게시 방식이 없습니다.**  
   정적 사이트에는 관리자 서버가 없습니다. 초기에는 내부 운영자가 DB 관리 화면이나 제한된 관리 스크립트로 게시하는 방식이 가장 단순합니다. 공개 테이블에 anon 사용자가 직접 게시하도록 하면 안 됩니다.

10. **사용자 제출과 공개 게시 데이터를 분리해야 합니다.**  
    사용자 제출을 허용한다면 `submissions`에만 INSERT하고, 검수된 결과만 `board_items`로 옮겨야 합니다. 그렇지 않으면 스팸·악성 링크·허위 공고가 바로 노출될 수 있습니다. 서버 없이 익명 제출을 열면 요청량 제한도 어렵기 때문에 MVP에서는 제외하는 편이 안전합니다.

11. **RLS 공개 범위가 정의되지 않았습니다.**  
    anon 사용자는 `published` 상태의 공개 필드만 읽어야 합니다. 내부 메모, 수집 원문, 검수자 정보, 삭제된 게시물은 공개 조회 결과에 포함되면 안 됩니다.

12. **중복 게시물 처리 기준이 없습니다.**  
    하나의 지원사업이 기업마당·주관기관·보도자료에 동시에 등장할 수 있습니다. 원문 URL 정규화, 제목·기관·마감일 조합 등을 이용한 중복 후보 판정이 필요합니다.

13. **공식 원문과 재배포 기사의 우선순위가 없습니다.**  
    지원사업·공모전은 뉴스 기사보다 주관기관 공식 공고를 대표 출처로 삼아야 합니다. 보조 출처는 별도로 연결하는 편이 정확합니다.

14. **저작권 정책이 더 구체적이어야 합니다.**  
    “제목+요약”만으로 충분하지 않습니다.

    - 요약은 원문 문장 복사 대신 직접 작성
    - 포스터·썸네일은 MVP에서 사용하지 않음
    - 이미지 주소를 그대로 연결하는 핫링크도 금지
    - 출처별 이용약관·API 조건·자동 수집 허용 여부 확인
    - 삭제·수정 요청을 받을 연락처와 처리 절차 마련

15. **검색 로봇 규칙은 수집 허가가 아닙니다.**  
    자동 수집 전 각 사이트의 이용약관, API 사용 조건, 호출 제한을 확인해야 합니다. 법적 판단이 필요한 소스는 MVP 대상에서 제외하는 편이 낫습니다.

16. **요약 생성의 오류 가능성이 빠졌습니다.**  
    자동 요약이 지원 자격·금액·마감일을 잘못 바꾸면 위험합니다. 마감일·대상·금액 같은 핵심 사실은 원문 필드에서 구조화하고, 자동 요약 결과는 검수 전 공개하지 않는 기준이 필요합니다.

17. **외부 링크 보안이 없습니다.**  
    URL 스킴을 `https/http`로 제한하고, 악성 주소와 리디렉션을 검사해야 합니다. 수집한 HTML을 그대로 렌더링해서도 안 됩니다.

18. **장애와 빈 상태가 구분되지 않습니다.**  
    Supabase 조회 실패를 “게시물 없음”으로 보여주면 안 됩니다. 로딩·데이터 없음·조회 실패·게시물 삭제 상태를 각각 처리해야 합니다.

19. **뉴스와 지원사업은 같은 운영 모델이 아닙니다.**  
    뉴스는 양이 많고 저작권 위험이 높으며 수명이 짧습니다. 지원사업은 양이 적지만 정확도가 중요합니다. 처음부터 한 수집기로 묶으면 복잡도만 커집니다.

20. **개인 폰트의 범위가 여전히 불명확합니다.**  
    “새로운 폰트 출시 소식”은 게시판에 둘 수 있지만, 검색 가능한 폰트 메타데이터는 #64가 소유해야 합니다. 게시판은 #64의 폰트 ID를 선택적으로 참조하되 원본 정보를 복제하지 않는 편이 좋습니다.

21. **유료 폰트는 광고·거래 경계가 필요합니다.**  
    외부 판매처 링크만 제공할지, 제휴 링크나 유료 홍보를 허용할지 정해야 합니다. 유료 홍보가 들어오면 광고 표시와 노출 기준까지 별도 정책이 필요합니다.

22. **카테고리만으로 분류가 부족할 수 있습니다.**  
    지원사업과 공모전이 겹치는 경우가 있습니다. 하나의 고정 카테고리와 함께 `지원 대상`, `지역`, `무료/유료` 같은 태그를 선택적으로 두는 편이 낫습니다.

23. **운영 품질 지표가 없습니다.**  
    체류·재방문 증가가 목적이라면 출처 클릭률, 재방문율, 게시물 최신성 등을 측정해야 합니다. 단순 게시물 노출은 목적 달성 여부를 증명하지 못합니다.

24. **#64와 코드·스키마는 분리하되 운영 기반은 재사용할 수 있습니다.**  
    테이블과 수집 로직은 분리하는 것이 맞습니다. 다만 OCI cron, 배치 계정, 로그·알림 방식까지 중복 구축할 필요는 없습니다.

### 개선 제안

1. **최우선 결정 사항**

   - MVP 카테고리: `지원사업` 하나
   - 운영 방식: 수동 큐레이션
   - 표시 방식: 정적 게시판 화면에서 Supabase 런타임 조회
   - 콘텐츠: 제목, 직접 작성한 짧은 요약, 공식 출처 링크만
   - 이미지·뉴스·사용자 제출·거래 기능: 제외
   - 상태: 마감일을 기준으로 런타임 계산하고 취소·조기 종료만 수동 지정

2. **MVP 슬라이스**

   - Slice 0: 후보 출처 2~3곳의 이용 조건·데이터 형식·갱신 빈도 확인
   - Slice 1: 운영자가 검수한 지원사업 게시물 등록
   - Slice 2: 목록, 카테고리, 마감 임박 정렬, 마감 상태 표시
   - Slice 3: 런타임 상세 조회와 원문 이동
   - Slice 4: 수정·취소·마감 연장·보관 운영 절차
   - Slice 5: 클릭률과 최신성 측정 후 자동 수집 투자 여부 결정

3. **MVP 완료조건 권장안**

   - 검수된 지원사업 게시물이 목록과 상세 화면에 표시된다.
   - 각 게시물에 공식 출처, 원문 게시일, fontagit 등록일, 마지막 확인일이 표시된다.
   - 마감일·시간대·마감 방식이 저장되고 진행 중·마감·취소 상태가 올바르게 표시된다.
   - 마감 임박 순 정렬에서 상시 모집과 일정 미정 항목이 별도로 처리된다.
   - Supabase 장애, 빈 목록, 삭제된 게시물 화면이 구분된다.
   - anon 사용자는 공개 게시물만 읽을 수 있고 게시·수정·삭제할 수 없다.
   - 원문 변경 시 운영자가 마감 연장·취소·내용 수정을 반영할 수 있다.
   - 본문 복제와 외부 이미지를 사용하지 않으며 삭제 요청 처리 경로가 표시된다.

4. **후속 우선순위**

   - 1순위: 지원사업 수동 큐레이션
   - 2순위: 공모전 추가
   - 3순위: 검수 전제의 반자동 수집·중복 탐지
   - 4순위: 개인 폰트 출시 소식과 #64 연결
   - 최후순위: 뉴스 자동 수집과 사용자 제출

5. **이슈 구조 권장안**

   #66을 바로 구현 이슈로 사용하지 말고 Epic으로 전환하는 것이 좋습니다. `출처·법적 검토`, `데이터 모델`, `운영자 등록`, `런타임 게시판`, `마감 상태`, `자동 수집`을 각각 하위 이슈로 분리해야 계획과 완료 여부를 검증할 수 있습니다.
