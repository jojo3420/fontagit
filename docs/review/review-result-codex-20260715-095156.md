### 구체적 피드백
1. [Blocker] 90% 판정 기준이 너무 주관적입니다.  
   “육안 diff”만으로는 구현자가 어디까지 고쳐야 하는지 애매합니다. 최소한 `레이아웃`, `모듈 존재`, `간격`, `타이포`, `텍스트`별 체크 항목을 만들어야 합니다.

2. [Blocker] 디자인 프레임과 실제 라우트 매핑표가 없습니다.  
   예: `1g -> /fonts/{free-slug}`, `6a -> /fonts/{paid-slug}`처럼 명확히 적어야 구현 계획에서 바로 작업할 수 있습니다.

3. 데이터 모델 검증 규칙이 부족합니다.  
   `유료면 priceFrom 필수` 외에도 `license.webfont`, `license.redistribution`, `verifiedAt`, `officialUrl` 누락 시 처리 방침이 필요합니다.

4. `officialUrl` 호스트 파생은 실패 케이스가 있습니다.  
   URL이 없거나 잘못된 경우, `www.` 제거 여부, 한글 도메인, 리다이렉트 등을 어떻게 처리할지 정해야 합니다.

5. 무료 대안 카드 조건이 애매합니다.  
   “최대 3개”는 좋지만, 0개/1개/2개일 때 카드 자체를 숨길지, 빈 상태를 보여줄지 정해야 합니다.

6. 유료 폰트 견본 정책이 더 구체적이어야 합니다.  
   실제 서체 미보유 시 어떤 대체 폰트를 쓰는지, 사용자에게 “실제 폰트가 아닐 수 있음”을 표시할지 정해야 합니다.

7. 모바일 기준이 부족합니다.  
   “1단으로 접힘” 외에 사이드바 순서, sticky 해제, CTA 위치, 카드 간격 기준이 필요합니다.

8. 다크모드 검증 범위가 애매합니다.  
   토큰은 비목표라고 했지만, 카드 테두리/버튼/안내 박스 대비가 실제로 깨질 수 있습니다. 최소 확인 항목은 필요합니다.

9. 슬라이스 2~8은 아직 너무 얕습니다.  
   “각 착수 시 상세화”는 괜찮지만, 전체 설계 스펙이라면 최소한 각 화면의 기준 라우트와 핵심 모듈은 더 적어야 합니다.

10. Header 차이 예외는 잘 적었습니다.  
   다만 90% 판정에서 “네비 6개 유지”가 어떤 항목에서 제외되는지 명확히 해야 합니다.

### 개선 제안
1. 우선순위 1: 화면별 매핑표를 추가하세요.  
   `디자인 프레임 / 실제 라우트 / 대표 데이터 / 뷰포트 / 완료 기준` 형식이면 충분합니다.

2. 우선순위 2: 90% 체크리스트를 수치화하세요.  
   예: 모듈 존재 40점, 레이아웃 30점, 간격/타이포 20점, 텍스트 10점.

3. 우선순위 3: 데이터 예외 정책을 문서에 넣으세요.  
   URL 없음, 가격 없음, 무료 대안 없음, 라이선스 확인일 없음 같은 경우의 UI 표시를 정해야 합니다.

4. 우선순위 4: 슬라이스 1만큼은 컴포넌트 입력값까지 정의하세요.  
   `LicenseSummaryCard`, `AlternativesCard`, `Breadcrumb`의 props 수준까지 적으면 구현 계획 품질이 좋아집니다.

5. 우선순위 5: 구현 계획 작성 전, 슬라이스 1 완료 기준을 더 엄격하게 잠그세요.  
   지금 문서는 방향성은 좋지만, 그대로 구현하면 “대충 비슷함”에서 논쟁이 생길 수 있습니다.
