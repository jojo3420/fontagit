### 동의하는 핵심 피드백 (Top 3)
1. **#28 수치 불일치(180≠200)** — 실행 착수 전 반드시 확정. 원 이슈 결함이므로 사용자가 A 분류를 120으로 하거나 20종 추가 분류를 정해야 한다.
2. **dev→prod 반영 흐름 + 롤백 부재** — prod가 라이브라 시드 데이터 승격 절차(백업→승인→적용→SSG 재빌드→화면 확인→복구)가 없으면 사고 시 되돌릴 수 없다.
3. **DB 시드 안전장치(멱등/dry-run/단일 세션)** — 두 세션이 동시에 시드하면 중복. 실행 전후 카운트 비교와 DB 쓰기 단일 세션 원칙이 필요하다.

### 동의하지 않는/과한 피드백
- **모든 이슈 동일 깊이 요구(#4 일부)**: 이 문서는 #62 전체 트래킹 로드맵이지 단일 이슈 상세 설계가 아니다. #55/#54/#34/#36은 "결정 대기"라 결정 전 상세화가 무의미. 완료조건-검증 최소치만 보강하면 충분.
- **"담당자" 반복 강조**: 담당이 사람이 아니라 여러 AI 세션이라 사람 담당자 개념은 맞지 않음. "DB 쓰기 단일 세션" 원칙으로 대체하는 게 정확.

### Codex가 놓친 추가 관점
- **dev DB 조회 도구(B1)가 실제 프로덕트 결함이 아니라 로컬 MCP 설정 문제일 가능성**: 시드 실행 자체는 MCP가 아니라 파이프라인 코드(`noonnu_import.py`)가 하므로, B1이 해소 안 돼도 파이프라인 직접 실행으로 시드는 가능하다. B1은 "검증 편의" 블로커이지 "시드 불가" 블로커는 아닐 수 있음 → 문서가 B1을 과대 차단으로 표현했을 여지.
