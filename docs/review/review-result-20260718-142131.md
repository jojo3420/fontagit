## 구체적 피드백

1. **[HIGH·실현가능성] 눈누 미리보기 웹폰트가 동적 서브셋일 가능성.** Tier B 1,110종 대부분은 공식 파일이 없어 눈누 `@font-face` 파일에 의존한다(Task 11 metadata stage). 그런데 눈누 미리보기 웹폰트는 페이지에 보이는 글자만 담은 서브셋일 수 있고, 그러면 cmap에 한글이 극히 일부만 있어 **거의 전량이 needs_review**로 빠진다. 설계·계획 어디에도 "서브셋 폰트 감지/제외" 방어가 없다. 이건 이번 변경의 핵심(cmap 판별)을 무력화할 수 있는 최대 리스크다.

2. **[HIGH·실현가능성] KS X 1001 2,350자 완전일치 기준이 지나치게 엄격.** `classify_scripts`는 2,350자 전체가 있어야 `korean` verified, 하나라도 빠지면 needs_review다(Task 11). 손글씨·제목용 디자인 폰트는 잘 안 쓰는 음절을 빠뜨리는 경우가 흔한데, 이들이 눈누 Tier B의 다수다. 결과적으로 정상 한글 폰트가 대량 needs_review로 몰려 파일럿 10% 게이트(Task 6/12)와 정면 충돌한다. "2,350 완전일치" 대신 "커버율 임계값(예: 2,350 중 N%)" 또는 "11,172 현대 한글 부분집합 판정"을 고려해야 한다. `hangul_glyph_count`를 이미 저장하므로 임계값 판정 근거는 있다.

3. **[HIGH·완성도] `subsets` 컬럼 정의 위치 불명.** design 5.1은 `subsets`를 신규 컬럼처럼 나열하고 0019 CHECK는 `cardinality(subsets)`를 참조하며 mapper도 읽는데, **Task 1의 0017 `add column` 목록에 `subsets`가 없다.** 메모리·design 2절은 "기본값 빈 배열로 남는다"고 해 기존 컬럼으로 보이지만, 기존 컬럼의 타입이 `text[] not null default '{}'`가 아니면 `cardinality()`와 판정이 깨진다. 착수 전 실제 스키마를 `rg`/`psql \d`로 확인하고, 없으면 0017에 추가해야 한다. 같은 맥락에서 `allow_modify`/`allow_redistribute`/`allow_embedding`의 값 도메인 CHECK가 0016 재사용에만 의존하는데, 0016이 이 도메인(`allowed`/`conditional`/`denied`)을 강제하는지 검증되지 않았다.

4. **[MED·일관성] 부분 한글일 때 확실한 라틴 정보까지 버린다.** `test_partial_hangul_requires_review`는 라틴 52자가 완전한데도 `subsets == []`를 기대한다. 라틴은 verified 조건을 이미 충족했으므로 `subsets=["latin"]` + `script_status=needs_review`(한글만 미확정)로 남기는 편이 정보 손실이 없고 검수·견본 선택에도 유리하다. 현재 방식은 "한글 미확정"과 "아무것도 확인 못함"을 구분 못 한다.

5. **[MED·일관성] UI `sourceMode` 단일값이 3축 독립 상태와 충돌.** download/license/script가 각각 pending·verified·needs_review·broken을 독립적으로 가지는데, mapper는 `sourceMode: "audit" | "legacy"` 하나로 판정한다(Task 9). license는 verified인데 download는 pending인 혼합 행에서 legacy/audit 판정이 모호해진다. 또 `broken`(다운로드 링크 죽음)에 "라이선스 재확인 필요"를 함께 붙이면(9절·Task 9) 라이선스가 verified여도 오해를 준다. 축별 표시를 분리해야 한다.

6. **[MED·실현가능성] 되돌리기 reverse manifest의 `updated_at` precondition이 불명확.** build_manifest가 forward/reverse를 동시에 생성하는데(Task 7), reverse의 `expected_updated_at`은 "정방향 적용 후" 값이어야 한다. 정방향 적용이 `updated_at`을 touch하면 reverse precondition이 어긋나 되돌리기가 0건 적용될 수 있다. 허용 필드 목록에 `updated_at`이 없으니 apply RPC가 이를 건드리지 않는다는 전제로 보이는데, 그렇다면 "감사 변경이 `updated_at`에 반영되지 않는다"는 부작용과 함께 문서에 명시하고, SQL 왕복 테스트에 `updated_at` 시나리오를 넣어야 한다.

7. **[MED·완성도] 0019 CHECK 강화 전 `pending=0` 강제 게이트가 없다.** 0019 `fonts_published_license_chk`는 published 행에 `license_status in (verified, needs_review)`만 허용한다. 조사 실패로 `pending` 행이 하나라도 남으면 0019 마이그레이션 전체가 실패(롤백)한다. Task 13 Step 6은 곧바로 0019를 적용할 뿐, 직전 `pending` 카운트 0 검증을 명시하지 않는다. script/download/license 세 축 모두 동일.

8. **[MED·누락] "여러 분할 파일 병합"의 판정 기준·테스트 부재.** unicode-range로 쪼갠 서브셋 파일은 각각 부분 cmap이라, "언제 합집합을 취하는가"가 판정 순서를 좌우한다. 설계는 "같은 family·굵기·스타일이면 합집합"이라 했으나, face 단위 판정인지 family 단위인지, 합집합 전 개별 파일을 먼저 classify하면 전부 needs_review가 되는지가 불명확하다. Task 11 테스트 3개는 모두 단일 cmap이고 합집합 케이스가 없다.

9. **[LOW·완성도] `KS_X_1001_HANGUL` 컴프리헨션에 디코드 예외 방어 없음.** `bytes((lead,trail)).decode("euc_kr")`는 0xB0A1~0xC8FE 전 범위가 유효해 실제로는 2,350자가 정확히 나오지만, 특정 셀이 매핑되지 않으면 **모듈 로드 시 크래시**한다. `errors` 처리나 명시적 유효성 검증을 두는 편이 안전하다(개수 == 2350 assert 포함).

10. **[LOW·일관성] broken 24시간 판정 로직이 두 곳에 중복.** `classify_download`(Task 4)와 `import_observations`(Task 10)가 각각 24h·2회 규칙을 구현한다. 단일 함수로 공유하지 않으면 향후 규칙 변경 시 불일치가 생긴다.

## 개선 제안 (우선순위)

**P0 — 착수 전 반드시 해소**
1. 눈누 웹폰트 서브셋 여부를 감지(예: cmap 크기·`OS/2` 서명·파일 크기 휴리스틱)해 서브셋 의심 파일은 판별 근거에서 제외하고 needs_review로 명시. 공식 배포 파일 확보 경로(제작사·공유마당)를 Tier B에도 우선 적용.
2. 한글 판정을 "2,350 완전일치"에서 "커버율 임계값 + `hangul_glyph_count` 근거 기록"으로 완화. 임계 미달만 needs_review로. 파일럿 10% 게이트가 현실적으로 통과되는지 50종에서 먼저 실측.
3. `subsets` 실제 스키마와 `allow_modify`/`redistribute`/`embedding`의 기존 CHECK를 `psql \d fontagit.fonts`로 확인 후, 없으면 0017/0019에 명시적으로 추가.

**P1 — 구현 중 반영**
4. 부분 한글 + 완전 라틴은 `subsets=["latin"]`으로 라틴 정보 보존.
5. UI를 download/license/script **축별 표시**로 분리하고, broken 문구를 "다운로드 링크 확인 불가"로 라이선스와 구분.
6. 0019 적용 직전 세 축 `pending=0`을 psql 집계로 강제 검증하는 게이트를 Task 13에 추가. 미충족 시 중단.
7. reverse manifest의 `updated_at` 처리를 문서에 명문화하고 SQL 왕복 테스트에 포함.

**P2 — 품질 보강**
8. 분할 파일 cmap 합집합 케이스를 테스트에 추가하고 face/family 판정 단위를 명확화.
9. 24h broken 판정을 단일 공유 함수로 통합, `KS_X_1001_HANGUL`에 개수 assert 추가.

**다음 단계**: P0 3건은 실측(50종 파일럿 dry-run + 실제 눈누 파일 1~2개 cmap 확인)으로 리스크를 먼저 검증한 뒤, 판정 임계값과 스키마를 확정하고 계획 문서를 갱신할 것을 권한다.
