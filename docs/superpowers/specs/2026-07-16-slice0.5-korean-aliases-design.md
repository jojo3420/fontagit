## 1. 배경 — 왜 필요한가

슬라이스2 검색 계획을 구현 착수 전 dev 실측으로 검증한 결과, 검색의 핵심 가치를 실증할 데이터가 없음을 확인했다.

**dev 실측 (2026-07-16, `zgxtfcpiokhkcrywlxmc`)**
- published 폰트 130종, aliases 272개 — 데이터는 있으나
- `name_ko` 있는 폰트 **0건**, 한글 별칭 **0건**
- aliases 272개는 전부 기계 생성된 영문명 변형(`anton`, `antonttf`)
- 정규화 검증: alias_norm에 공백-대문자 잔존 0건 → 파이프라인 규칙(공백제거+소문자)과 검색 `normalize_search` 일치 확인
- pg_trgm 미설치(슬라이스2 마이그레이션이 설치 예정)

**원인**: `transform.py:150`의 `to_record`가 `build_aliases(raw.family)`만 호출해 `name_ko`를 넘기지 않는다. `build_aliases`(43행)는 `name_ko` 인자를 받을 준비가 돼 있으나 호출부가 안 넘긴다. 또한 구글폰트 API에는 한글 이름이 없어(`family`는 "Noto Sans KR" 같은 영문 표기) 한글 데이터는 반드시 외부 소스가 필요하다.

계획대로 슬라이스2를 바로 구현하면 완료 기준("지마켓/지머켓 별칭-오타 매칭")을 실데이터로 증명할 수 없다. 사용자 결정: **한글 별칭 적재 선행**.

## 2. 범위

- 대상: korean subset **38종**(현재 published 130종 중). 현재 라틴 전용 published 목표는 92종이며 `name_ko`는 null을 유지한다.
- 넣을 것: `name_ko`(한글 공식/통용 이름) + 근거 있는 통용 별칭(예: Noto Sans KR → 본고딕).
- 안 넣을 것: 근거 없는 창작 별명(정직성 원칙).
- **name_ko 작성 기준**: 공식 한글 표기(제작사 또는 구글폰트의 한글 표기)가 확인될 때만 기입, 없으면 **null**(음역 창작 금지). 웹 `mappers.ts`가 `name_ko ?? name_en` 폴백이라 null 안전.
- **라틴 정합의 정의**: 구글 인기순 변동으로 published 집합은 바뀔 수 있다. DB published Tier A 집합은 현재 파이프라인 결과와 완전히 같아야 하며, 재적재 전후 공통 slug의 alias/alias_norm 집합은 동일하고 name_ko는 null이어야 한다. 빠진 원본은 삭제하지 않고 draft로 보존한다.

38종: noto-sans-kr, nanum-gothic, nanum-myeongjo, noto-serif-kr, gothic-a1, nanum-gothic-coding, black-han-sans, nanum-pen-script, jua, dongle, do-hyeon, nanum-brush-script, gowun-batang, ibm-plex-sans-kr, gaegu, hahmlet, gamja-flower, gowun-dodum, sunflower, hi-melody, asta-sans, bagel-fat-one, poor-story, dokdo, song-myung, gugi, gasoek-one, east-sea-dokdo, stylish, yeon-sung, cute-font, single-day, orbit, kirang-haerang, diphylleia, black-and-white-picture, moirai-one, grandiflora-one.

## 3. 결정 사항 (뒤집지 않음 — 변경 시 사용자 확인)

| 항목 | 결정 | 근거 |
|---|---|---|
| 데이터 소스 | 웹검색 근거 큐레이션 | 정확-정직, 근거 출처 기록 |
| 별칭 범위 | 이름(name_ko) + 통용 별칭 | 실사용 검색어 커버(본고딕 등), 근거 있는 것만 |
| 조사 실행 | 병렬 서브에이전트(general-purpose + WebSearch) | 공개정보 웹검색은 위임 허용으로 해석 |
| 적재 경로 | 파이프라인 재업로드(`upsert_font`, 멱등) | 파이프라인이 SSoT, dev 직접 삽입 우회 금지 |
| stale 폰트 | 전체 성공 후 현재 스냅샷에 없는 Tier A published를 draft 전환 | 현재 원본과 DB published 집합 일치, 행·slug 보존 |
| 적용 환경 | dev만, **prod 미적용** | 핸드오프 제약: prod DB 쓰기 금지 |

## 4. 설계

**4.1 큐레이션 매핑 파일** `apps/pipeline/src/fontagit_pipeline/data/korean_names.json` (패키지 내장 — 모듈 기준 경로, hatchling wheel 포함)
```
{ "noto-sans-kr": { "name_ko": "노토 산스 KR", "aliases": ["본고딕", "노토산스"], "sources": ["<근거 URL 1개 이상>"] }, ... }
```
- 38 slug. 각 항목에 `sources`(근거 출처 URL 배열, 1개 이상) 기록.
- `aliases`는 통용 별칭만(name_ko 자동 파생형은 build_aliases가 생성하므로 중복 불필요).
- `sources`는 큐레이션 근거 전용 — FontRecord에 해당 필드가 없어 DB/웹으로 유입되지 않음(노출 경계).

**4.2 transform 통합**
- 매핑 로드 함수 추가(파일 부재/파싱 실패는 경계 검증). 경로는 모듈 기준 고정(`Path(__file__)` 상대 또는 importlib.resources) — 실행 위치(CWD)와 무관.
- `build_aliases(name_en, name_ko=None, extra_aliases=None)` — 시그니처에 `extra_aliases` 추가, 기존 중복제거-순서유지 로직 재사용.
- `to_record`: slug로 매핑 조회 → `name_ko` 설정 + `build_aliases(family, name_ko, extra)` 호출. 라틴(korean subset 아님) 폰트는 기존 동작 유지.
- **fail-fast**: korean subset 폰트가 매핑에 없거나, 매핑에 대상 외 잉여 slug가 있으면 transform 단계에서 즉시 에러(침묵 실패 금지).

**4.3 정규화 SSoT**
- 한글 별칭도 업로더 `normalize_alias`(공백제거+소문자) 경유 저장. 한글은 소문자 무효지만 공백제거는 적용 → "본 고딕"→"본고딕". 검색 `normalize_search`와 동일 규칙이라 정확매칭 정합.
- **유니코드 NFC**: 큐레이션 입력에 NFD(자모 분리형) 혼입 가능 → 정규화 순서를 **NFC → 공백제거 → lower**로 명세. Python은 `unicodedata.normalize("NFC", ...)`, 슬라이스2 `normalize_search`도 동일 순서로 구현(교차 검증은 슬라이스2 계획의 정규화 SSoT 테스트 항목 활용).

**4.4 재적재**
- **전량 검증 후 쓰기**: 매핑 스키마 검증-38종 대조-빈값 검사와 레코드 변환을 전부 끝낸 뒤에만 DB 쓰기 시작(부분 실패 최소화). 중간 실패 시 멱등 재실행으로 복구(`upload_records` 기존 설계: 첫 실패 즉시 중단, 처리분 유지).
- 파이프라인 실행 → dev `upsert_font`(폰트 upsert + aliases 원자 교체). 멱등.
- 전체 upsert 성공 뒤에만 `sync_tier_a_fonts` 실행. 일반 부분 업로드와 전체 스냅샷 업로드 인터페이스를 분리한다.
- 안전장치: active Tier A 고유 slug 100종 미만이면 첫 쓰기 전 중단, stale published 후보 5종 초과면 draft 갱신 전 중단. Tier B/C는 제외한다.
- 검증(메인 직접, psql): `has_hangul_alias > 0`, 샘플 폰트(noto-sans-kr 등) 별칭에 한글 포함 확인.

**4.5 테스트**
- `build_aliases`: name_ko + extra_aliases 병합, 중복제거, 순서유지.
- 매핑 파일 스키마: 38 slug 존재, 필드 타입, source 필수.
- transform 통합: 매핑 유(name_ko/aliases 반영)/무(영문만) 분기.
- 정규화 일관성: 한글 별칭 norm 규칙.

## 5. 완료 기준

- 매핑 파일 38종 작성(각 근거 출처 URL 1개 이상 포함).
- 파이프라인 재적재 후 dev 전수 검증: 한글 별칭 보유 폰트 수 = 매핑에서 한글 별칭을 가진 폰트 수, name_ko 반영 수 = 매핑의 name_ko 비-null 수(1종 성공만으로 통과 금지). 대표 폰트(noto-sans-kr 등) 별칭 샘플 확인.
- 테스트 통과(로그 첨부). DB published Tier A와 현재 파이프라인 결과의 slug 완전 일치, 공통 라틴 slug 내용 무변경, 라틴 name_ko null을 확인.
- 정직성: 근거 없는 별칭 0.

## 6. 이후 연결

이 슬라이스 완료 후 슬라이스2(검색 F-04)로 복귀. 그때 아래 3건 반영:
- **테스트 케이스 실데이터 교체**: 슬라이스2 계획의 완료 기준-테스트-예시가 지마켓 산스 전제이나 Gmarket Sans는 구글폰트에 없어 이 데이터셋에 존재할 수 없음 → 실데이터 기반(노토산스/본고딕/나눔고딕 + 오타 변형)으로 교체.
- 버그 B: `50 - (similarity(...) * 0)::int` → trgm 점수 항상 50 고정. 실제 유사도 반영으로 수정.
- 버그 C: 유사도 대상이 `fonts.name_en`인데 GIN trgm 인덱스는 `aliases.alias_norm` → 대상을 `alias_norm`으로 교정(인덱스 활용 + 별칭 오타 검색 동작).
